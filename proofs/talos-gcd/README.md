# Talos Proofs

This workspace contains Lean proofs about selected WASM modules generated by LeanExe.  LeanExe compiles a Lean source entry to WASM, `wasm-tools print` renders that artifact as WAT, and Talos decodes the generated WAT into a Lean representation of the WASM module.  The handwritten `Spec.lean` files prove semantic properties of that decoded module under Talos's WASM model.

The proof input remains tied to the compiler output through the scripts in [`tools`](../../tools).  Each script regenerates WASM and WAT from the Lean source, compares those files against the checked-in proof inputs, and then rebuilds the relevant Lean proof.  A file mismatch means the proof no longer describes the current LeanExe output for that example.

Talos's verifier emitter generates each `Program.lean` file from the matching `program.wat`.  Those generated files should be regenerated from the artifact rather than edited by hand.  The `Spec.lean` files are the proof sources that name the generated functions, state the expected behavior, and discharge the generated control-flow and memory obligations.

## Requirements

The Lean proof project pins Talos through the `CodeLib` Lake dependency at revision `bb3277e21c9786e3133d5c1601e34ebdc0bea4df`.  The check scripts need `wasm-tools` to render WAT from regenerated WASM.  They look for `WASM_TOOLS`, then `wasm-tools` in `PATH`, then `$HOME/.cargo/bin/wasm-tools`.

Run an individual proof check from the repository root:

```sh
tools/check-talos-gcd.sh
tools/check-talos-assoc-list.sh
tools/check-talos-order-book.sh
```

Run every current proof check:

```sh
tools/check-talos.sh
```

## Current Proofs

| Case | Lean source | Export | Theorem | Scope |
|------|-------------|--------|---------|-------|
| GCD | [`LeanExe.Examples.TalosGcd`](../../LeanExe/Examples/TalosGcd.lean) | `gcd` | `Project.Gcd.Spec.gcd_correct` | For all `a b : UInt64`, the generated export terminates and returns `UInt64.ofNat (Nat.gcd a.toNat b.toNat)`. |
| Association-list lookup | [`LeanExe.Examples.TalosAssocList`](../../LeanExe/Examples/TalosAssocList.lean) | `lookupDemo` | `Project.AssocList.Spec.lookupDemo_correct` | For every `UInt64` key, the generated export returns the first matching value from the source-level sample list, or `0` when absent. |
| Order-book matching | [`LeanExe.Examples.OrderBook`](../../LeanExe/Examples/OrderBook.lean) | `matchBook` | `Project.OrderBook.Spec.matchBook_correct` | For all seven scalar inputs encoding one best bid, one best ask, and one incoming order, the generated export returns the expected option tag, quantity, and price. |
| Byte validation | [`LeanExe.Examples.AsciiDigits`](../../LeanExe/Examples/AsciiDigits.lean) | `validateGeneric` | `Project.Validate.Spec.validateGeneric_correct` | For every byte list, every store holding those bytes at a pointer, and every pointer, the generated export terminates and returns `1` exactly when all bytes are ASCII digits. |

The association-list proof has a fixed source-level list because the exported program constructs that list internally before searching it.  The theorem is still universal over the program input, the lookup key.  The lookup function is specified against an abstract `ListSegAt` predicate describing a linked association-list segment in memory, so the search lemma is one induction over the list, holds for arbitrary constructed segments, and contains no concrete cell addresses; only the lemma connecting the constructed sample to the predicate mentions the concrete allocation.  The order-book proof does not fix the book or the order; the one-level book and incoming order are supplied through scalar ABI slots.

Lemmas shared between artifact proofs live in [`lean/Project/Common.lean`](lean/Project/Common.lean): `UInt64` arithmetic under `omega`, the conversions between wrapped 32-bit addresses and their modular normal forms, and indexed list access.

The byte-validation proof is the first over the memory model with universally quantified input.  Its `BytesAt` hypothesis states what the host wrote into linear memory at the input pointer, the loop invariant carries the scan position and the all-digits-so-far fact, and the shared `ValidateSpec` from the Lean source instantiates both the plain-Lean theorem about `validateGeneric` and the Talos theorem about the decoded artifact.

## Check Scripts

Each per-case script wraps [`tools/check-talos-case.sh`](../../tools/check-talos-case.sh), which performs the artifact-integrity check.  It builds the Lean source module and the `lean-wasm` executable, compiles the selected source entry to a temporary WASM file, renders temporary WAT with `wasm-tools print`, compares both temporary files against the checked-in proof inputs, and rebuilds the relevant `Spec.lean` file.  The script stops at the first mismatch or failed proof.

The checked-in files under `rust/build/<case>/program.wasm` and `rust/build/<case>/program.wat` are the proof inputs.  The generated `Program.lean` file is derived from the checked-in WAT by Talos's verifier emitter.  When the compiler output changes intentionally, run the corresponding check script with `--update`:

```sh
tools/check-talos-order-book.sh --update
```

Update mode replaces the proof inputs with fresh compiler output, regenerates the matching `Program.lean` through the verifier emitter, and rebuilds the proof.  A proof failure after an update means the `Spec.lean` file needs repair before committing.  `tools/check-talos.sh --update` updates every case.  Update mode expects the verifier emitter binary at `lean/.lake/packages/CodeLib/verifier/.lake/build/bin/verifier`.

The `emit` case name matches the directory under `rust/build`.  The generated Lean file is `lean/Project/OrderBook/Program.lean` for `order_book`, `lean/Project/AssocList/Program.lean` for `assoc_list`, and `lean/Project/Gcd/Program.lean` for `gcd`.

## Proof Boundary

These proofs establish properties of selected generated WASM artifacts after Talos decodes the generated WAT.  Their scope is the decoded artifact for one example.  The general LeanExe compiler-correctness theorem and a proof relating Talos semantics to Wasmtime remain outside this workspace.

The artifact comparison closes a narrower gap.  It ensures that the proof input matches the WASM and WAT generated by the current Lean source and compiler for a particular case.  Wasmtime runs remain useful execution checks for selected inputs, but the universal statements in this workspace come from the Talos proofs.

The broader compiler-correctness goal remains the theorem described in [`plan.md`](../../plan.md): accepted Lean source, extracted IR, lowered WASM, and the modeled WASM execution should compose into one source-to-WASM correctness result.  The proofs here are end-to-end artifact experiments that exercise the generated code path while that general theorem remains future work.
