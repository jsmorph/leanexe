# Talos Proofs

This workspace contains Lean proofs about WASM modules generated by LeanExe.  LeanExe compiles a Lean source entry to WASM, `wasm-tools print` renders that artifact as WAT, and Talos decodes the generated WAT into a Lean representation of the WASM module.  The handwritten `Spec.lean` files prove semantic properties of that decoded module under Talos's WASM model.

The two proof tools regenerate the selected artifact and decoded model before Lean checks a specification.  WASM, WAT, and `Program.lean` remain ignored local outputs, while the source, case registration, and handwritten proof modules remain tracked.  Talos's Cargo-shaped input exists only in a temporary directory created during model generation.

[Verifying a Program](../../docs/verifying.md) defines every stage, input, output, and failure boundary.  It also explains registration, runtime pins, proof structure, and the aggregate gate.  The development guide supplies the repository-wide process limits and test requirements.

## Architecture

Every generated module ends with the same four runtime functions: allocate, reset, retain, and release.  They are byte-identical across modules except that release embeds its own function index at its two recursive call sites.  [`lean/Project/Runtime/Defs.lean`](lean/Project/Runtime/Defs.lean) names those instruction streams once, while [`lean/Project/Runtime/Checks.lean`](lean/Project/Runtime/Checks.lean) pins every module's functions to the shared definitions by `rfl`.

[`lean/Project/Runtime/Spec.lean`](lean/Project/Runtime/Spec.lean) states the runtime's behavior generically over the module and the function index, with a lookup hypothesis each artifact discharges by `rfl`: the exported retain, the null release, the shared-object decrement, and the raw-object free.  An artifact proof consumes these through the call rule instead of re-proving them.

[`lean/Project/Runtime/Tree.lean`](lean/Project/Runtime/Tree.lean) and [`lean/Project/Runtime/TreeSpec.lean`](lean/Project/Runtime/TreeSpec.lean) carry the recursive case.  A `RelTree` models an ownership tree of slots-kind heap objects at refcount one whose masked slots hold null, an owned subtree, or a shared object; `TreeAt` ties the tree to a memory through the reads the release walk performs.  The generic teardown theorem, `release_frees_tree`, proves that releasing the root frees every owned node in traversal order, decrements every shared leaf, leaves the free list at the root, and advances the release and free counters by exactly the tree's event counts.  A new program's recursive teardown proof reduces to exhibiting the tree shape, footprint disjointness, and a page bound.

[`lean/Project/Common.lean`](lean/Project/Common.lean) holds the shared mechanical layer: frame lemmas for reads over writes, the `read_frames` tactic that resolves a read over a write chain by peeling disjoint frames to the hit, `UInt64` and address-form bridges under `omega`, and the `BytesAt` predicate stating what the host wrote into linear memory at an input pointer.

Two statement templates cover the artifacts.  Input-generic theorems quantify over the program's inputs and relate the result to the Lean source function: `gcd` returns `Nat.gcd`, `foldSum` returns the source fold, `validateGeneric` accepts exactly the all-digit strings.  Counter theorems pin the runtime's accounting for a concrete allocation story: exact release and free counts, free-list order, and untouched memory below the old heap top.  Both shapes state frame conditions, so callers know their memory survives.

## Layout

| Path | Purpose |
|------|---------|
| [`lean/Project.lean`](lean/Project.lean) | Imports every checked proof spec in this workspace. |
| [`cases.json`](cases.json) | Registers source entries, generated module names, proof targets, and completion state. |
| `.generated/<case>/program.wasm` and `program.wat` | Ignored current compiler output retained for local inspection. |
| `lean/Project/<Case>/Program.lean` | Ignored Talos-generated Lean model.  Regenerated, never edited. |
| `lean/Project/<Case>/Spec.lean` | Tracked handwritten proof entry point for that artifact. |
| [`lean/Project/Runtime`](lean/Project/Runtime) | The shared runtime: instruction streams, `rfl` pins, module-generic lemmas, and the teardown theorem. |
| [`lean/Project/Common.lean`](lean/Project/Common.lean) | Frame lemmas, tactics, and arithmetic bridges shared by all specs. |

## Workflow Tools

[`talos-artifact.js`](../../tools/talos-artifact.js) builds the registered source module and compiler, emits WASM, renders WAT, and asks Talos to generate `Program.lean`.  It stages the complete result in a temporary directory and replaces local generated outputs only after every stage succeeds.  It generates the minimal Cargo metadata required by Talos in that temporary directory and removes it before returning.

[`talos-proof.js`](../../tools/talos-proof.js) always performs artifact generation before building a selected handwritten proof.  Its aggregate mode generates all twenty registered models, verifies the registry against runtime and specification imports, and builds the nineteen completed specifications through `Project`.  Both tools run every Lean-based child under the required memory, CPU, scheduling, I/O, and timeout limits.

```sh
tools/talos-artifact.js prepare gcd
tools/talos-proof.js check gcd
tools/talos-proof.js check --all
```

## Requirements

The proof and compiler workspaces pin Lean 4.31.0 in their respective `lean-toolchain` files.  The proof Lake manifest pins Talos and its transitive dependencies.  The artifact tool fetches that pinned dependency and builds its verifier under the resource limits when a local verifier is absent.

The tools require `wasm-tools` 1.251.0 to render WAT from regenerated WASM.  They look for `WASM_TOOLS`, then `wasm-tools` in `PATH`, then `$HOME/.cargo/bin/wasm-tools`, and reject another version before generating an artifact.  [Developing LeanExe](../../DEVELOPING.md) gives the complete setup, environment variables, process policy, and failure diagnostics.

```sh
tools/talos-artifact.js prepare --all
tools/talos-proof.js check --all
```

Artifact generation stages a complete case before replacement.  A generation failure leaves the prior local WASM, WAT, and generated model unchanged, while a replacement or cleanup failure makes the command fail and names the affected path.  Handwritten proof files never participate in replacement and therefore remain untouched.

## Proof Boundary

These proofs establish properties of selected generated WASM artifacts after Talos decodes the generated WAT.  Their scope is the model freshly derived from the current source and compiler during the proof gate, under Talos's WASM semantics.  Connecting a separately distributed binary with that theorem also requires the same pinned artifact tools or a byte comparison with the gate's generated WASM.

The general LeanExe compiler-correctness theorem covers accepted Lean source, extracted IR, lowered WASM, and modeled WASM execution in one source-to-WASM result.  That theorem remains the destination described in [Development Plan](../../plan.md).  The runtime lemma library reduces the per-program work by replacing repeated instruction proofs with applications of shared theorems.
