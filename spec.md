# LeanExe Language Specification

## Purpose

LeanExe defines a restricted executable subset of Lean 4 for extraction to WebAssembly.  Lean remains the parser, elaborator, type checker, and proof checker.  The compiler consumes checked declarations from a Lean environment and rejects declarations outside the supported subset.

The subset targets traditional pure programming first: validators, parsers, encoders, finite decision procedures, and deterministic byte-oriented transforms.  The design also reserves a path for modeled effects, including a restricted `IO` fragment.  Effect support must enter the core IR explicitly, because an extracted Wasm module cannot inherit Lean’s runtime behavior without making that runtime part of the trusted base.

## Source Boundary

The compiler reads checked Lean declarations from a Lean environment.  It does not parse Lean syntax, elaborate terms, resolve overloads, or trust source text.  A declaration enters the compiler only after Lean has accepted the module that contains it.

An entry point is a named Lean constant with a closed type after specialization.  The compiler computes the runtime-relevant dependency graph from the entry point’s checked value and type.  Proof-only dependencies may be erased, but executable dependencies must either compile or receive an exact rejection reason.

The current report imports compiled `.olean` modules through Lean’s module loader.  It expands declarations whose root namespace matches the imported module root and records other dependencies as an external frontier.  The frontier prevents the report from recursively expanding Lean’s standard library and distinguishes implemented primitives from dependencies that still block generic extraction.

`Nat` needs careful treatment because Lean’s `Nat` is unbounded, while the first Wasm target uses fixed-width machine integers and explicit memory.  The subset may admit `Nat` in proofs, sizes, indices, and compile-time constants before it admits general unbounded runtime arithmetic.  Any accepted runtime `Nat` operation must state its representation and overflow behavior.

Runtime `Nat` values in the current fragment use the same `i64` representation as other scalar values.  Runtime `Nat` literals must be below `2^64`; larger literals are rejected rather than truncated unless they are consumed directly by a fixed-width conversion such as `Nat.toUInt64`.  `Nat` subtraction follows Lean’s saturating semantics: `a - b` returns `0` when `a < b`.  `Nat.succ` uses the checked addition path, and `Nat.pred` uses the saturating subtraction path.  `Nat.toUInt64` preserves the bounded runtime representation and lowers large literals modulo `2^64`.  `Nat` addition and multiplication trap when the result exceeds the bounded representation.  `Nat` division and remainder use Lean’s zero-divisor behavior: `x / 0` returns `0`, and `x % 0` returns `x`.  `Nat.min`, `Nat.max`, `Nat.beq`, `Nat.blt`, and `Nat.ble` lower through unsigned comparisons or equality over the bounded representation.  Programs that rely on arbitrary-precision runtime `Nat` results are outside the current subset.

`UInt32` is an internal scalar type in the current fragment.  The generic fragment represents it as an `i64` constrained to `0..2^32-1`, and exported entries still reject `UInt32` parameters and results.  `UInt32` literals and `UInt32.ofNat` lower modulo `2^32`, `UInt32.toNat`, `UInt32.toUInt64`, and `UInt8.toUInt32` preserve the constrained representation, and `UInt64.toUInt32` masks to 32 bits.  `UInt32.toUInt8` masks to eight bits.  `UInt32` addition, subtraction, and multiplication mask their result to 32 bits.  `UInt32` division and remainder use Lean’s zero-divisor behavior: `x / 0` returns `0`, and `x % 0` returns `x`.  `UInt32` left shifts mask the shift amount modulo 32 and then mask the result to 32 bits; right shifts mask the shift amount modulo 32.

The first accepted term language should resemble a first-order functional IR rather than Lean’s full expression language.  It should contain variables, lets, calls, constructors, projections, case analysis, loops, and primitive byte and integer operations.  Each accepted Lean construct must translate to that IR without consulting Lean runtime objects at execution time.

Local `let` support preserves Lean’s lexical de Bruijn scope with lazy extractor bindings.  A let-bound expression is extracted only when the body demands it, so `let x := bad; (x, y).2` returns `y` without evaluating `bad`, matching Lean evaluation.  Let-bound values may use nested lets, arrays, branches, calls, products, structures, source-defined inductives, and `Option` values when their types remain inside the first fragment.  Product and `Option` values exist only inside the extractor; product projection and `Option` matching lower by selecting demanded scalar expressions, and the Wasm ABI still exposes only `i64` scalar or pointer values for those internal values.

Product construction follows Lean’s projection behavior: projecting one field does not force extraction or evaluation of the unused field.  This matters for partial terms such as `(bad, x).2`, where Lean returns `x` without evaluating `bad`.  Product pattern matching supports ordinary generated matchers and direct `Prod.casesOn` applications, and the extractor recognizes `Prod.rec` if a checked term contains it.  The matcher arm receives the fields as lazy internal values, so unused fields do not contribute to the emitted Wasm expression.  Product-valued entry parameters and product-valued entry results are rejected until the public ABI assigns products source type identity.

User-defined structures use Lean structure metadata for source identity and field order.  The current extractor accepts structures only when the inductive has no parameters, no indices, one constructor, no recursion, and runtime fields that map to supported types.  Constructor applications build extractor-level structure values, projection functions and `.proj` terms select fields lazily, structure update syntax works when Lean elaborates it to projections plus a constructor, and single-constructor matches bind fields lazily in source field order.  Structure entry parameters and exported structure results flatten in Lean field order after removing proof fields; nested structures flatten recursively, and a supported fixed-width array field occupies one `i64` pointer slot.  Polymorphic structures, recursive structures, inherited-field flattening, and unsupported runtime fields remain unsupported.

User-defined inductives use Lean constructor and recursor metadata for source identity, constructor order, and runtime field layout.  The current extractor accepts inductives only when the type has no parameters, no indices, is not a structure, is not recursive, and every non-proof constructor field maps to a supported type.  Constructor applications build extractor-level tagged values.  Generated matchers and direct recursors bind constructor fields in source order; proof binders receive an erased placeholder, and runtime binders receive lazy payload values.  Nullary constructor arms may elaborate as `Unit` lambdas and lower through the same erased-unit path used for `Bool` arms.

User-defined inductive entry parameters and results use a fixed tagged multi-value ABI.  The first slot is the constructor tag, using the constructor's declaration index.  The remaining slots contain the flattened payload fields for every constructor in declaration order; inactive constructor payload slots in results receive default payload values.  A nullary enum therefore uses one `i64` tag, while `inductive Status | ok : UInt64 -> Status | error : UInt64 -> Status` uses three `i64` slots: tag, `ok` payload slot, and `error` payload slot.  `Option α` uses the same rule: tag first, followed by the flattened `some` payload slots.  `Except ε α` uses tag, flattened error payload slots, and flattened success payload slots.  Recursive inductives, indexed inductives, polymorphic inductives, proposition-valued matcher motives, and arrays of inductives remain unsupported.

`Option` uses the source-identified tagged-value path.  Tag `0` represents `none`, with no constructor payload fields.  Tag `1` represents `some`, with one payload field.  Matching an `Option` value lowers to a conditional over the tag, and only the selected branch contributes to the emitted scalar expression.  `Option.getD`, `Option.get!`, `Option.orElse`, `Option.elim`, `Option.map`, `Option.filter`, `Option.any`, `Option.all`, `Option.bind`, `Option.isSome`, and `Option.isNone` lower through the same tag representation.  `Option.getD` and `Option.elim` evaluate default expressions only for `none`, `Option.get!` traps for `none`, `Option.orElse` evaluates the fallback thunk only for `none`, `Option.map`, `Option.filter`, `Option.any`, `Option.all`, and `Option.bind` evaluate their functions only for `some`, and tag tests do not evaluate a `some` payload.  The payload of `some bad` is not evaluated when the `some` arm ignores it, and the `some` arm is not evaluated for `none`.  `Option` entry parameters and results can cross the current Wasm ABI when their payload fields can be flattened.

`Except` uses the source-identified tagged-value path.  Tag `0` represents `Except.error`, with one error payload field.  Tag `1` represents `Except.ok`, with one success payload field.  Pattern matching evaluates only the selected arm, and each arm receives its payload as a lazy internal value.  `Except.map` and `Except.bind` evaluate their functions only for `ok`.  `Except.mapError` evaluates its function only for `error`.  `Except.toOption` preserves the tag and success payload while discarding the error payload.  `Except.isOk` reads only the tag.  The `<|>` fallback form evaluates the fallback thunk only for `error`.  `Except Unit α` works because `Option` and `Except` now have separate source identities.  `Except` entry parameters and results can cross the current Wasm ABI when their payload fields can be flattened.

Nonrecursive project-local helper calls are inlined into the caller with lazy argument thunks.  Helpers may use supported internal local types, including `UInt8`, `UInt32`, products, `Option`, and `Except`, even though exported entries cannot expose several of those types in the current ABI.  Strict project-local calls that remain as Wasm calls use demand summaries before their arguments are emitted.  For each helper, the extractor records parameters that the helper may demand and parameters that it must demand when its result is demanded; argument trap analysis expands project-local helper calls through the may-demand summary.  A strict call is rejected when an argument may trap and the callee does not must-demand that parameter.  Recursive summaries are conservative: the fuel parameter is must-demanded, carried parameters are may-demanded, and more precise carried-parameter demand remains planned.

Boolean conjunction and disjunction lower with Lean-compatible short-circuiting.  The emitted Wasm must not evaluate the right-hand side of `true || rhs` or `false && rhs`, because the skipped expression may contain a partial operation such as an out-of-bounds array access.  Proposition-level `And`, `Or`, and `Not` lower for supported conditions with the same branch behavior.  `LT.lt`, `LE.le`, `GT.gt`, and `GE.ge` lower to unsigned `i64` comparisons for `UInt64` and for bounded `Nat` values already admitted into the fragment.  Scalar equality propositions over `Bool`, `UInt8`, `UInt64`, and bounded `Nat` lower to scalar equality.  `Decidable.decide` lowers supported decidable comparisons, scalar equalities, and proposition connectives to `Bool`.  `UInt64` division and remainder follow Lean’s checked behavior at zero divisors: `x / 0` returns `0`, and `x % 0` returns `x`.

The supported recursive loop shape has one decreasing `Nat` fuel parameter followed by first-fragment carried values.  The base arm compiles to the result used when fuel reaches zero.  The successor arm may either tail-call the same recursive handle, use `if cond then exitValue else recursiveCall`, or use `if cond then recursiveCall else exitValue`.  In the conditional cases, the emitted loop continues only while fuel remains and the recursive branch is selected, then returns `exitValue` when the source condition selects the exit branch before fuel reaches zero.  The extractor rejects non-tail recursion and successor arms whose recursive call is not one branch of that immediate conditional shape.

## Structures and Results

Structure support uses Lean's checked structure metadata as the source of truth for source identity.  Lean 4.29 records structure information through `StructureInfo`, `getStructureInfo?`, `getStructureFieldsFlattened`, `StructureInfo.getProjFn?`, and projection-function metadata in the environment.  The current implementation uses constructor and projection metadata for monomorphic, nonrecursive structures, and it requires the constructor field list to match the flattened field list without inherited subobject expansion.

The implemented structure subset accepts runtime fields that belong to the supported first fragment.  Runtime fields may include scalar values and supported fixed-width array pointers at the public ABI, while nested supported structures flatten recursively.  A constructor field is erased when its checked field type is a proposition by direct sort or by a fully applied constant whose declared result is `Prop`; erased fields are omitted from the runtime structure value, public result list, and Wasm ABI.  Recursive structures, unspecialized polymorphic structures, dependent runtime fields, function fields, effectful fields, inherited-field flattening, and opaque field producers remain unsupported.

Internal representation records source type identity through `Ty.struct name fields` and a matching extractor value that stores the structure name and ordered runtime fields.  This prevents unrelated shapes from being conflated and lets projections select by Lean projection identity.  Constructor applications build structured values, projections select fields lazily, structure match arms receive lazy field bindings, and structure update syntax compiles when Lean elaborates it to projections plus a constructor.

Returned structure values use a flat multi-value Wasm ABI.  A returned structure flattens its public runtime fields in Lean field order, recursively flattening nested supported structures; each scalar, bounded `Nat`, `Bool`, and supported fixed-width array field occupies one `i64` result.  `ByteArray` results and fields that require owned byte output remain planned, because the current `ByteArray` support is a read-only view into host-provided memory and does not define output ownership.  Reports show the source result type, so a source structure return such as `Point` remains visible as `Point` even when the exported Wasm function returns two `i64` values.

## Arrays

The current generic fragment represents each supported array as an `i64` byte offset into Wasm linear memory.  The pointer addresses an array header.  Offset `0` stores the element length as `i64`.  Element payload cells begin at offset `8`.  A scalar element has width one, so element `i` is stored at byte offset `8 * (i + 1)`.  A multi-slot element with width `w` stores slot `s` of element `i` at byte offset `8 * (1 + i * w + s)`.

Scalar arrays support `Bool`, `UInt8`, `UInt32`, `UInt64`, and bounded `Nat` elements.  `Bool` cells use `0` or `1`, `UInt8` and `UInt32` cells use their constrained scalar representations, and `Nat` cells use the bounded `i64` representation.  Structure arrays flatten each runtime field in Lean field order after proof-field erasure.  Tagged arrays store the constructor tag first, followed by flattened payload slots for every constructor in declaration order, matching the public tagged ABI.  Inactive payload slots use the default values produced by constructor extraction.

Array literals lower when Lean elaborates them as `List.toArray` over a literal list whose item type has a fixed width; general `List` values remain unsupported.  `Array.empty`, `Array.mkEmpty`, and `Array.emptyWithCapacity` allocate an empty array and do not evaluate the capacity argument because capacity is not observable in the current layout.  `Array.singleton v` allocates one element and writes every slot of `v`.  `Array.replicate n v` currently supports only scalar element arrays; it allocates the header and `n` eight-byte cells from a module-global bump pointer, and nonzero replication emits a fill loop.

`Array.size` lowers to a header load, and `Array.isEmpty` compares that size with zero.  `GetElem.getElem`, the elaborated form of proof-indexed `a[i]`, erases the proof argument and lowers to a checked element load.  `Array.get!Internal`, `GetElem?.getElem!`, and `Array.back!` lower to bounds checks followed by slot loads for the demanded element fields.  If the index is out of bounds, the emitted Wasm executes `unreachable`, matching the panic behavior of ordinary Lean execution for `a[i]!` and empty `back!`.  Safe indexing through `GetElem?.getElem?`, the elaborated form of `a[i]?`, evaluates the array and index once, returns `some` with a loaded element when the index is in bounds, and returns `none` without loading demanded payload fields when the index is out of bounds.  `Array.back?` evaluates the array once, returns `none` for an empty array, and returns `some` with the last element otherwise.  Proof-indexed `Array.back` erases its proof argument and lowers to the same last-element read as `back!`; the Lean proof establishes that the empty-array trap path is unreachable in accepted source programs.

`Array.set` erases its proof argument and lowers to the same copy-on-write replacement operation as an in-bounds `set!`.  `Array.setIfInBounds` evaluates the array and index once, evaluates the replacement value only when the index is in bounds, returns the original array when out of bounds, and otherwise allocates a fresh array, copies every payload cell, and writes every replacement slot.  `Array.set!` evaluates its array, index, and value arguments, checks the index, allocates a fresh array, copies every payload cell, writes every replacement slot, and returns the new pointer; if the index is out of bounds, it traps.  `Array.swapAt` erases its proof argument, returns an internal product of the old element and the copy-on-write updated array, and preserves product projection behavior: projecting `.1` evaluates the array and index without evaluating the replacement value.

`Array.push`, `Array.pop`, `Array.append`, `Array.extract`, bounded insertion, bounded erasure, bounded swaps, and `Array.reverse` operate over payload-cell counts for fixed-width arrays.  Insertion copies prefix cells, writes the inserted element slots, and copies suffix cells to their new offset.  Erasure copies prefix cells and then copies the suffix over the erased element.  Swapping copies the full payload and rewrites each slot of the two selected elements.  Reversal reverses element order while preserving slot order inside each element.

`Array.getD` loads a fixed-width element when the index is in bounds and selects the default value only when the index is out of bounds.  `Array.modify` loads the old element as a structured value, evaluates the modification function only for an in-bounds index, flattens the returned element, and lowers the update through the same copy-on-write replacement path as `setIfInBounds`.

The remaining array operation using the older one-cell lowering is `map`.  `Array.replicate` also remains scalar-only because multi-slot replication needs an explicit rule for evaluating the replicated value once and copying all slots into each element.  Existing aliases continue to observe the old array for every implemented copy-on-write update.

Resizing, polymorphic arrays, nested arrays, and arrays of recursive types are unsupported.  The copy-on-write lowering is semantically conservative and slow.  A future linear array optimization must have a checker that rejects aliasing patterns before using in-place update.

## Byte Arrays

The generic fragment represents an entry `ByteArray` parameter as two Wasm `i64` parameters: a byte pointer and a byte length.  Inside the extractor, that pair remains one structured value, so a Lean function still has one `ByteArray` parameter and helper calls receive the same source-level value.  `ByteArray.size` returns the length slot, `ByteArray.isEmpty` compares that length with zero, and `ByteArray.get! input index`, proof-indexed `input[index]`, and `input[index]!` lower to a bounds check followed by `i32.load8_u` and zero extension to the fragment’s scalar `i64` representation.  Safe indexing through `input[index]?` evaluates the pointer, length, and index once, returns `some` with an internal `UInt8` when the index is in bounds, and returns `none` without loading when the index is out of bounds.  `ByteArray.extract input start stop` evaluates the source, start, and stop once, clamps stop to the source length, returns an empty slice when the effective stop is not greater than start, and otherwise returns a read-only pointer-length view into the original bytes.

`ByteArray.get!` produces an internal `UInt8` value.  The generic fragment represents that value as an `i64` constrained to `0..255`.  `UInt8` literals and `UInt8.ofNat` lower modulo `256`, matching Lean’s checked semantics, and `UInt8.toNat` is representation-preserving inside the bounded `Nat` fragment.  `UInt8` addition, subtraction, and multiplication mask their result to eight bits.  `UInt8` left shifts mask the shift amount modulo eight and then mask the result to eight bits; right shifts mask the shift amount modulo eight.  `UInt8` division and remainder use the same zero-divisor behavior as Lean: `x / 0` returns `0`, and `x % 0` returns `x`.

The current support is read-only.  Hosts write input bytes into exported memory before calling the entry function; generic modules export `alloc(len)` and `reset()` so the host can place byte inputs in the same bump arena used by compiled array allocation.  `LeanExe.Examples.AsciiDigits.validateGeneric : ByteArray -> Bool` exercises this path through `lean-wasm compile`; the older `validate` declaration remains the proof-oriented source for the hand-written validator path.

| Panic or partial operations | Rejected unless eliminated or modeled | Array bounds failures are modeled as Wasm traps; general panic remains unsupported |
| Lean `IO` | Planned later as an explicit capability subset | Reported only |
| File access | Planned later under explicit host imports | Reported only |
| Environment variables | Planned later under explicit host imports | Reported only |
| Time and randomness | Planned later only with modeled nondeterminism | Reported only |
| Concurrency | Rejected for the first effectful subset | Rejected |
| Arbitrary FFI | Rejected | Rejected |

Initial `IO` support should be small and explicit.  A future effectful IR should represent operations such as reading a file, writing bytes, reading an environment variable, or returning an error as operations with declared host imports.  The compiler must reject hidden runtime access, ambient nondeterminism, and capabilities that the Wasm module signature does not declare.

## Dependency Classification

The extractor classifies each reachable declaration from the entry point.  Runtime declarations contain executable content that affects the output value.  Proof declarations inhabit propositions or erased fields and may appear in the checked environment without entering the Wasm program.

The report must name every rejected declaration and the first unsupported construct found in it.  Useful rejection reasons include opaque constant, unsupported primitive, unsupported type, higher-order value escape, unspecialized polymorphism, typeclass argument that cannot be specialized, unsupported recursor, unsupported effect, `unsafe` declaration, and runtime environment dependency.  A rejection reason is part of the user-facing API and should remain stable enough for tests.

The current report implements module loading, entry lookup, root-namespace dependency expansion, external frontier classification, entry-shape classification from the generic compiler signature checker, and effect detection for `IO`, `EIO`, `BaseIO`, and `Task`.  It classifies the implemented scalar, product, structure, user-defined inductive, `Option`, `Except`, array, byte-array, pattern-matching, and conversion primitives used by the current compiler fragment.  Proof declarations in the expanded namespace do not contribute runtime dependencies to the external frontier.  The report still classifies polymorphic declarations, typeclass instance dependencies, target-dependent numeric literals, and unimplemented library operations as pending extraction work.  It rejects `unsafe`, `partial`, opaque executable constants, axioms in executable dependency graphs, quotients, higher-order arguments, unsupported effects, and external constants without a LeanExe primitive.

## Current Wasm ABI

The current validator module exports one linear memory, `alloc`, `reset`, and `validate`.  The host writes input bytes into memory, calls `validate(ptr, len)`, and receives `0` or `1`.  The arena begins at byte offset `4096` and resets per call.  Generic CoreWasm modules export one 16-page linear memory, `alloc(len : i64) -> i64`, `reset()`, and the requested entry function.  Their arena also begins at byte offset `4096`.  Entry short names `memory`, `alloc`, and `reset` are reserved by the runtime ABI and are rejected.  Scalar values, booleans, bounded `Nat` values, and array pointers cross the ABI as `i64`, while a `ByteArray` parameter crosses as two `i64` values, `ptr` and `len`.  Structure results use Wasm multi-value results by flattening supported public fields in Lean field order.  User-defined inductive, `Option`, and `Except` results return a tag followed by flattened payload slots for every constructor in declaration order.

Flattened parameters keep source order: a `ByteArray` parameter contributes its `ptr` and `len` slots at its source position, and scalar parameters stay in their source positions around those two slots.  Structure parameters flatten their runtime fields in Lean field order after proof-field erasure.  Tagged parameters use the same slot order as tagged results: tag first, followed by flattened payload fields for each constructor in declaration order.  The byte-array allocation harness tests byte-array parameters before and after scalar parameters.  Structured outputs remain planned for `ByteArray` and parser result buffers, because each needs ownership rules.  Host imports must remain explicit in the Wasm module.

## Correctness Obligations

The compiler should maintain separate correctness claims for extraction, proof erasure, specialization, lowering, and Wasm emission.  The current repository proves a small source-to-core equality for the demo validator and a lowering lemma for the byte-range validator.  The first generic compiler fragment does not yet prove source-to-IR equivalence.  The repository tests the emitted Wasm against Lean evaluation through direct Wasmtime runs, generic ByteArray WAST checks, and the validator differential harness.

Future proofs should compose toward `compile_correct`, relating the modeled Wasm execution to the checked Lean function for accepted entry points.  The final theorem should state the trusted base: Lean’s kernel and admitted axioms, the formal semantics of the modeled Wasm subset, the emitter if unverified, the host ABI, the Wasm runtime, and the underlying platform.  Browser, Wasmtime, or Wasmer execution should not be described as formally verified unless the proof includes that runtime.
